## 1. 本手册的用途

本手册用于在**网站主体交付之后**，向甲方说明：

- 实际已交付的能力与页面范围（含超出最初约定的增量）  
- 技术栈、部署方式与环境依赖  
- 日常内容/营销相关的**维护入口**（改哪里、谁负责）  
- **上线后服务**的时间边界、会议节奏、变更请求方式  

更细的技术与页面索引见仓库内：`SITE_MAP.md`、`PROJECT.md`、`CLAUDE.md`、`TOUR_UPDATE_INSTRUCTIONS.md`。

## 2. 实际已交付范围（相对最初需求的扩展说明）

下列能力在迭代中已陆续上线，整体规模**显著大于「单站首页 + 基础目录」类最小交付**。若未来合同或月费需对齐工作量，建议以本清单为**范围基线**。

### 2.1 产品与行程

- 全站行程目录、目的地分层（Signature / Discovery / Stopover）、动态行程详情页  
- 行程打印页（`/print` 路线）与邮件行程请求（`/api/send-itinerary`，依赖 Resend，见第 6 节）  
- 行程数据集中维护：`src/lib/data/tours.ts`（字段含行程、价格展示、图库、JSON-LD 等）

### 2.2 SEO 与内容矩阵

- **Phase 1 枢纽页**（示例）：`/china-tours`、各目的地 `*-tours` 枢纽、`/china-tours-from-new-zealand`、`/china-tours-from-auckland`、`/best-time-to-visit-china`、`/china-visa-guide-for-new-zealanders` 等（完整列表见 `SITE_MAP.md`）  
- **Phase 2 目的地与景点指南**（21+ 深度内容页），与行程、枢纽页互链  
- 站内站点图页面：`/site-map`（若已启用）及 `SITE_MAP.md` 中的链接架构说明  
- 十月战役等**活动落地页**：`/campaigns/october-2026` 及子路径；与 Discovery 产品 URL 的规范跳转（见 `SITE_MAP.md` 与 `next.config.js` 中的 308 规则）

### 2.3 体验与性能相关

- 图片迁移至 Supabase Storage 的工作流与大量资源上线（历史记录见 `CLAUDE.md` / 变更说明）  
- 导航与关键按钮的**加载状态**反馈，减少「点击无反应」感知问题  

### 2.4 内部营销与战役运营工具（非对客主站路径）

- `/marketing` — 营销中枢占位与扩展位  
- `/marketing/campaign` — 2026 战役看板（可选密码保护，环境变量 `MARKETING_PLAN_ACCESS_KEY`）  
- `/marketing/campaign/social` — 社媒发帖计划（数据驱动，源码数据文件见该页说明）  
- 旧路径 `/marketing-plan` 若配置则 **308** 至 `/marketing/campaign`  

上述工具面向**内部运营/协作验收**，不替代对客 SEO 页面；访问控制与保密约定由双方另行确认。

### 2.5 其他已存在能力（按仓库路由，可能持续增减）

- 博客、本地美食指南、旅行工具、专家页、`/admin` 图片管理等（以线上实际导航与 `src/app` 为准）。  

**小结：** 当前交付是「**商业站点 + 大规模 SEO 内容 + 战役落地 + 内部营销看板**」的组合；后续若增加**固定频次内容制作、广告投放执行与复盘**，应视为**增量服务**，在时间与合同上与「网站维护」区分。

## 3. 技术栈与部署（甲方需知的最低信息）

| 项目 | 说明 |
|------|------|
| 框架 | Next.js 14（App Router）、TypeScript、Tailwind CSS |
| 主机 | Render（监听 GitHub `main`，合并后自动构建部署） |
| 媒体 / 数据 | Supabase（Storage 等）；具体 bucket 与路径以项目脚本与 `CLAUDE.md` 为准 |
| 邮件 | Resend；**必须在服务端按需初始化**，API Key：`RESEND_API_KEY`（未配置时邮件功能不可用） |
| 分析 | GA4 配置说明见仓库 `GA4_SETUP.md`；付费/社媒外链请按该文档使用 UTM |

**环境变量（示意，非完整列表）：**  
`NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY`、`RESEND_API_KEY`、可选 `MARKETING_PLAN_ACCESS_KEY`。  

生产环境变量由**托管方（Render）**在控制台配置；变更后通常需重新部署生效。

## 4. 内容维护：甲方与承建方的分工建议

| 类型 | 典型操作 | 建议负责方 | 代码/数据位置（主要） |
|------|-----------|------------|------------------------|
| 行程文案、天数、价格展示、图库 | 改具体产品 | 双方确认后由承建方改库并上线 | `src/lib/data/tours.ts`（详见 `TOUR_UPDATE_INSTRUCTIONS.md`） |
| 指南页正文、季节与签证表述 | 法务/事实变更需甲方确认 | 甲方提供修订要点，承建方落版 | 各 `src/app/.../page.tsx` 或关联数据模块 |
| 战役日期、标题、CTA | 营销节点变更 | 甲方确认事实，承建方改页面/数据 | `src/app/campaigns/...`、相关 data 文件 |
| 社媒排期表 | 增删帖、改状态、改链接；UTM 与示例链接 | 运营在约定流程内提出 | `src/lib/data/social-posting-plan-2026.ts`；规范与可复制示例见 `/marketing/campaign/social` 页顶区块 |
| 图片替换（Unsplash / 自拍图规范） | 换 hero、换图库 | 按项目约定的工作流执行 | 见 `CLAUDE.md` 图片工作流；Supabase 路径需一致 |

**签证与入境表述：** 网站内容须避免「保证入境」类承诺；以官方最新规则为准，关键页宜保留「请核实最新政策」类提示（与社媒发帖前检查清单一致）。

## 5. 上线后服务约定（承建方时间边界）

以下为**承建方自愿意投入**的节奏说明，便于甲方规划会议与需求批次；**若与已签署合同不一致，以合同为准**，本手册可作为续约或补充协议的附件。

| 项目 | 约定 |
|------|------|
| 每周时间 | 约 **10–15 小时** 用于本项目（含沟通、实施、轻量测试） |
| 例会 | **双周会**（建议固定周几与时段；议程提前 24h 由甲方或双方列出） |
| 需求入口 | 邮件或双方确认的协作工具；**紧急定义**（例如影响下单或全站不可用）单独约定响应时间 |
| 默认不包含（除非另签） | 7×24 值守、无上限广告代投、每日发帖代运营、大量实拍与修图、法律咨询、甲方内部多部门并行催办 |

**超出每周 10–15 小时或超出「维护 + 约定增量」范畴的工作**，建议采用：**追加固定包（如每月 +N 小时）** 或 **项目制报价**，避免范围无限膨胀。

## 6. 已知限制与待办（交付诚实清单）

- **邮件行程：** 依赖 `RESEND_API_KEY`；未在生产配置前，API 不可用或失败。  
- **部分行程数据：** 历史上曾标注「发布前需与甲方核对完整性」；应以最新 `tours.ts` 与业务确认为准。  
- **营销看板密码：** 若启用 `MARKETING_PLAN_ACCESS_KEY`，须使用足够强度的密钥并限制知悉范围。

## 7. 变更与发布流程（简版）

1. **提出：** 甲方说明目标 URL、期望上线时间、是否涉及价格/法律敏感信息。  
2. **评估：** 承建方评估是否在当周时间盒内；超出则给补充报价或排期至下一迭代。  
3. **实施：** 在分支或直接按仓库流程修改 → 本地或 CI 构建通过 → 合并 `main`。  
4. **验证：** Render 部署完成后抽查关键页与表单；涉及追踪的链接检查 UTM 与 GA4。  
5. **记录：** 重要变更可写入 `CHANGELOG.md` 或双方共享的发布说明。
